# 项目周期上下文管理平台 MVP 架构指导文档

## 1. 产品定位

本项目不是 K8s 管理平台，也不是传统 Wiki，而是一个面向私有化交付 / 运维 / 项目团队的 **项目周期上下文管理平台**。

核心目标是：

* 以项目为中心管理所有历史文档。
* 以时间线方式追踪项目生命周期。
* 支持多人协作文档编辑。
* 支持类似 Notion 的全库搜索。
* 支持 AI 基于项目历史文档进行总结、检索、问答、交接文档生成。
* 支持项目级权限、文档权限、审计和版本追溯。

核心对象：

```text
项目 Project
文档 Document
文档 Block
时间线 Timeline
版本 Version
搜索索引 Search Item
AI 上下文 Context Pack
审计 Audit Log
```

```text

TypeScript
```

---

### AI 层

```text
Vercel AI SDK
厂商模型 API / 中转接口
自研 Agent Tools
轻量 Mastra Workflow
```

AI 层分成两类：

#### 实时 AI

用于 Plate 编辑器内：

* 续写
* 改写
* 总结选区
* 生成段落
* 生成变更记录
* 生成故障复盘模板

实时链路建议：

```text
Plate AI Plugin
  ↓
Hono /api/ai/editor-command
  ↓
Vercel AI SDK streamText
  ↓
厂商模型 API
  ↓
流式返回给 Plate
```

#### 项目知识 Agent

用于项目上下文问答：

* 总结项目最近变更
* 总结某次故障完整上下文
* 生成项目交接文档
* 提取项目风险
* 根据时间线生成阶段总结

Agent 链路建议：

```text
Plate / Chat UI
  ↓
Hono /api/ai/project-agent
  ↓
Vercel AI SDK + 自研 Tools
  ↓
Project Knowledge Tools
  ↓
PostgreSQL / Search / Vector
  ↓
流式返回
```

#### 轻量 Mastra 使用范围

Mastra 不作为主业务框架，只用于后台 AI workflow：

```text
文档保存后自动摘要
文档分块
embedding 生成
搜索索引更新
项目时间线摘要
项目周报生成
风险扫描
```

不建议 Mastra 直接负责：

* 文档协作
* 文档写入
* 权限主逻辑
* 项目 CRUD
* 审计事实源

## 5. 文档存储设计

文档不要只存一种形态。建议分为 4 层：

```text
Yjs Snapshot
  用于协作恢复

Plate JSON
  用于文档结构、版本、导出

Plain Text / Blocks
  用于全文搜索

Chunks / Summary / Embedding
  用于 AI 检索和上下文构建
```

核心原则：

```text
Yjs 管协作
Plate JSON 管结构
document_blocks 管搜索定位
search_items 管全库搜索
document_chunks 管 AI 上下文
```

## 6. 核心数据库表

### 6.1 projects

```sql
create table projects (
  id uuid primary key,
  tenant_id uuid not null,

  name text not null,
  customer_name text null,

  owner_id uuid null,

  description text null,
  tags text[] not null default '{}',

  created_by uuid not null,
  updated_by uuid not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz null
);
```

```sql

  id uuid primary key,
```

### 6.4 document_versions

用于版本追溯。

```sql
create table document_versions (
  id uuid primary key,
  document_id uuid not null references documents(id),

  version_no int not null,

  plate_json jsonb not null,
  plain_text text not null,
  html text null,

  change_summary text null,

  created_by uuid not null,
  created_at timestamptz not null default now(),

  unique(document_id, version_no)
);
```

版本生成时机：

```text
用户主动保存版本

每天自动快照
文档状态变更
重要记录提交
```

### 6.5 document_blocks

用于 Notion 类全文搜索和跳转定位。

```sql
create table document_blocks (
  id uuid primary key,

  tenant_id uuid not null,
  project_id uuid not null,
  document_id uuid not null references documents(id),

  block_id text not null,
  block_type text not null,

  path int[] not null,
  heading_path text[] not null default '{}',

  text_content text not null,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  unique(document_id, block_id)
);
```

用途：

```text
搜索命中文档具体段落
点击搜索结果跳转到对应 block
AI 可以直接拿 block 作为上下文
```

---

### 6.6 search_items

用于类似 Notion 的全库搜索。

```sql
create table search_items (
  id uuid primary key,

  tenant_id uuid not null,
  project_id uuid null,

  entity_type text not null,
  entity_id uuid not null,

  document_id uuid null,
  block_id text null,

  title text not null,
  subtitle text null,
  content text not null,
  path_text text null,

  doc_type text null,
  event_type text null,

  tags text[] not null default '{}',
  metadata jsonb not null default '{}',

  updated_at timestamptz not null default now()
);
```

entity_type：

```text
project
document
document_block
timeline_event
comment
```

MVP 搜索方式：

```sql
create extension if not exists pg_trgm;

create index idx_search_items_tenant
on search_items (tenant_id);

create index idx_search_items_project
on search_items (project_id);

create index idx_search_items_entity_type
on search_items (entity_type);

create index idx_search_items_title_trgm
on search_items using gin (title gin_trgm_ops);

create index idx_search_items_content_trgm
on search_items using gin (content gin_trgm_ops);
```

第一阶段搜索用：

```text
PostgreSQL + ILIKE + pg_trgm
```

后续增强：

```text
PostgreSQL FTS
PGroonga / pg_jieba
pgvector
Meilisearch / OpenSearch
```

不要一开始上 Elasticsearch。

## 7. 搜索设计

### 搜索目标

实现类似 Notion 的搜索：

```text
一个搜索框
  ↓
搜项目、文档、文档段落、时间线事件
  ↓
只返回当前用户有权限的数据
  ↓
展示标题、路径、摘要、命中内容
  ↓
点击跳转到文档具体 block
```

### 搜索入口

```text
GET /api/search?q=xxx
GET /api/projects/:projectId/search?q=xxx
```

### 搜索逻辑

优先查 `search_items`：

```text
project
document
document_block
timeline_event
```

排序规则 MVP：

```text
标题命中 > 路径命中 > 正文命中
当前项目 > 其他项目
最近更新 > 老文档
故障/变更类文档适当加权
```

搜索结果结构：

```json
{
  "items": [
    {
      "entityType": "document_block",
      "entityId": "uuid",
      "projectId": "uuid",
      "documentId": "uuid",
      "blockId": "block_xxx",
      "title": "客户 A 生产故障复盘",
      "path": "客户 A 项目 / 故障复盘 / 问题原因",
      "snippet": "排查发现 Helm values 中 callbackTimeout 配置未同步",
      "updatedAt": "2026-06-25T10:00:00Z"
    }
  ]
}
```

## 8. 文档写入和索引流程

### 协作写入流程

```text
用户编辑 Plate 文档
  ↓
Yjs 产生 update
  ↓
Hocuspocus 同步给其他用户
  ↓
Hocuspocus 保存 Yjs snapshot
  ↓
触发 document_indexing_job
```

### 后台索引流程

```text
document_indexing_job
  ↓
读取 Yjs snapshot / Plate JSON
  ↓
转换为 Plate JSON
  ↓
抽取 plain_text
  ↓
抽取 document_blocks
  ↓
更新 search_items
  ↓
生成 document_chunks
  ↓
生成 document summary
  ↓
可选生成 embedding
```

不要在每次输入时更新搜索索引。

建议触发策略：

```text
10~30 秒 debounce
用户离开文档
Hocuspocus onStoreDocument
手动保存版本
文档状态变更
```

## 9. AI Agent 设计

### Agent 不直接访问数据库

Agent 只能调用受控工具。

```text
Agent
  ↓
Project Knowledge Tools
  ↓
Permission Guard
  ↓
Domain Service
  ↓
Database / Search / Vector
```

---

### MVP Project Knowledge Tools

第一阶段建议实现这些：

```text
resolveProjectContext
getProjectProfile
getProjectTimeline
searchProjectDocuments
getDocumentReadableContent
getDocumentVersions
buildContextPack
createTimelineDraft
```

---

### Tool 标准结构

```ts
type ToolContext = {
  userId: string
  tenantId: string
  projectId?: string
  documentId?: string
  role: 'viewer' | 'editor' | 'admin'
  requestId: string
}
```

每个 Tool 必须：

```text
有 Zod input schema
有权限校验
有返回大小限制
有审计日志
有错误处理
不能越权读取
不能直接写入核心数据
```

---

### buildContextPack

这是 AI 系统的核心能力。

```text
输入：
项目 ID
问题主题
时间范围
模式：incident / handover / risk / weekly_summary / general

输出：
项目基础信息
相关时间线
相关文档
相关 block
摘要
证据来源
token 控制后的上下文包
```

Context Pack 示例：

```json
{
  "project": {
    "id": "p_123",
    "name": "客户 A 私有化项目"
  },
  "topic": "最近一次生产故障",
  "timeline": [],
  "documents": [],
  "evidence": [],
  "compressedSummary": ""
}
```

AI 最终回答必须基于 Context Pack。

## 11. 权限设计

最少需要这些角色：

```text
viewer
editor
admin
auditor
```

权限控制分三层：

```text
前端：
控制 UI 是否可编辑

Hocuspocus：
控制 viewer 只能接收 update，不能提交 update

API / Tools：
所有数据读取和 AI tool 调用必须做权限校验
```

不要只靠前端 `editor.setEditable(false)`。

## 15. 总结

本项目 MVP 的核心架构原则是：

```text
Plate 做编辑体验
Hocuspocus 做协作同步
Hono 做业务 API
PostgreSQL 做事实源和 MVP 搜索
search_items 做 Notion 类全库搜索
document_blocks 做文档内命中定位
Vercel AI SDK 做流式 AI 调用
Project Knowledge Tools 做受控知识访问
Mastra 只轻量用于后台 workflow
```

最重要的长期资产不是某个 Agent 框架，而是：

```text
项目知识结构
文档 block 结构
时间线结构
搜索读模型
Context Pack 构建能力
权限和审计体系
```

第一版只要把这些基础打稳，后续可以自然扩展到：

```text
语义搜索
向量检索
项目自动总结
风险扫描
交接包生成
跨项目知识发现
知识图谱
更复杂的 Agent workflow
```
